Confirmation & Safety Gates — Human-in-the-Loop Design

Exam: Claude Architect Foundation Chapter: 03 — Tool Use & MCP Difficulty: Intermediate Last Updated: 2026-06

📖 What Is This Topic About?

Powerful agents can take powerful actions — deleting records, sending communications, processing payments. Without appropriate safeguards, agents can cause irreversible harm even when following instructions correctly. This chapter covers how to design confirmation workflows, safety gates, and two-step execution patterns to keep humans appropriately in control.

🧠 Key Concepts

  • Safety Gate — A checkpoint that pauses agent execution and requires human approval before a consequential action proceeds
  • Two-Step Pattern — Separating a tool into a preview step (shows what will happen) and an execute step (performs the action)
  • Confirmation Token — A single-use, time-limited identifier generated by the preview step; the execute step requires this token to proceed
  • Dry Run — A parameter that runs a tool in simulation mode, showing what would happen without actually performing the action
  • Least Privilege — Designing agents to have only the permissions needed for their tasks; no more
  • Application Layer Enforcement — Business rules that must be enforced by the application code, not by prompts

📘 Detailed Explanation

Why Prompts Alone Are Not Enough for Safety

It might seem sufficient to instruct an agent: "Always ask for confirmation before deleting anything." But prompts are not reliable safety mechanisms:

PROBLEM WITH PROMPT-ONLY SAFETY:
"Always ask for confirmation before calling delete_file."

→ Works most of the time
→ But an agent may sometimes skip this under certain conditions:
  • The prompt is long and the instruction gets less attention
  • A user says "yes, I confirmed it, go ahead" and the agent proceeds
  • Complex multi-step tasks where the confirmation gets lost in context

SOLUTION: Enforce safety at the APPLICATION LAYER
→ The application physically cannot proceed without a valid confirmation token
→ No matter what Claude does or says, the execution step won't run without the token

Rule: Business-critical constraints must be enforced at the application layer, not through prompts.

The Two-Step Tool Pattern

Instead of a single tool that both previews and executes, split into two separate tools:

┌───────────────────────────────────────────────────────────────────┐
│                  TWO-STEP TOOL PATTERN                            │
│                                                                   │
│  STEP 1: PREVIEW                                                  │
│  ┌──────────────────────────────────────────────────────────┐    │
│  │  preview_remove_member(member_id)                        │    │
│  │                                                          │    │
│  │  Returns:                                                │    │
│  │  {                                                       │    │
│  │    "member_name": "Amit Sharma",                         │    │
│  │    "role": "Team Lead",                                  │    │
│  │    "active_projects": 3,                                 │    │
│  │    "consequence": "Will lose access to 3 active projects │    │
│  │                    and 47 documents",                    │    │
│  │    "confirmation_token": "tok_7f3a9b_exp_1719720000"     │    │
│  │  }                                                       │    │
│  └────────────────────────────┬─────────────────────────────┘    │
│                               │                                   │
│               Claude presents this preview to the user            │
│               User explicitly reviews and confirms                │
│                               │                                   │
│  STEP 2: EXECUTE (requires valid token)                           │
│  ┌──────────────────────────────────────────────────────────┐    │
│  │  execute_remove_member(member_id, confirmation_token)    │    │
│  │                                                          │    │
│  │  Application validates:                                  │    │
│  │  ✅ Token exists and hasn't expired                      │    │
│  │  ✅ Token was generated for THIS member_id               │    │
│  │  ✅ Token hasn't already been used                       │    │
│  │                                                          │    │
│  │  Only then: executes the removal                         │    │
│  └──────────────────────────────────────────────────────────┘    │
└───────────────────────────────────────────────────────────────────┘

Why confirmation tokens are better than a simple boolean:

  • A dry_run: false boolean can be passed by the agent in any call, bypassing the preview
  • A confirmation token is generated only by completing a valid preview — the execute step physically cannot run without it
  • Tokens are scoped to a specific action (removing THIS member, not any member)
  • Tokens expire, preventing stale confirmations from being replayed

Application Layer Enforcement — Hooks

For compliance-critical constraints (e.g., maximum refund amount, required approval for large transactions), the enforcement must happen in the application layer:

USER REQUEST: "Refund ₹15,000 to customer"
       │
       ▼
AGENT decides to call process_refund(amount=15000, customer_id="C001")
       │
       ▼  ← APPLICATION LAYER HOOK intercepts BEFORE execution
┌─────────────────────────────────────────────────────────────┐
│  COMPLIANCE HOOK                                            │
│  IF amount > ₹5,000:                                        │
│    BLOCK the tool call                                      │
│    INVOKE human escalation workflow                         │
│    Return error to Claude: "Amount exceeds autonomous limit │
│    of ₹5,000. Escalation triggered."                        │
└─────────────────────────────────────────────────────────────┘
       │
       ▼
Claude informs user that escalation has been triggered

This hook fires regardless of what Claude was instructed in the system prompt. The code enforces the rule; the prompt merely informs the agent.

Confirmation Workflow Design — What to Show Users

A confirmation request is only useful if the user can actually understand and verify what they're confirming. Bad confirmations look like:

❌ BAD CONFIRMATION:
"Are you sure you want to proceed? (yes/no)"
→ Users click yes without reading details
→ Mistakes happen with wrong accounts, wrong schedules, wrong content

✅ GOOD CONFIRMATION includes:
"Please review before publishing:
 📝 Content: 'Our Summer Sale starts June 20th — 30% off everything!'
 👤 Account: @certification_study_hub (3,400 followers)
 🕐 Scheduled: Monday 21 June 2026 at 9:00 AM IST
 📱 Platform: Twitter/X
 
 Type CONFIRM to publish or CANCEL to abort."

A good confirmation includes all the specific details (content, account, time, platform) so users can catch errors before they happen.

🇮🇳 Indian Real-Life Example

Think of the Two-Step Pattern like NEFT Bank Transfer:

When you make an online bank transfer:

  1. Preview step: The bank shows you: Beneficiary name, account number, IFSC code, amount, and says "Please verify these details before confirming"
  2. You review and click Confirm (generating a one-time OTP — the confirmation token)
  3. Execute step: You enter the OTP. The bank validates it and only then transfers the money.

Without the OTP (confirmation token), the transfer cannot proceed — no matter what the app interface shows. This is application-layer enforcement. The bank doesn't just trust that you reviewed the details — it makes confirmation cryptographically necessary.

🔑 Exam-Focused Points

  • ✅ Business-critical constraints must be enforced at the application layer, not just through prompts
  • ✅ The two-step pattern (preview tool + execute tool) prevents agents from bypassing confirmation
  • ✅ Confirmation tokens are better than dry_run: false because they:
    • Are generated only by completing a valid preview
    • Are scoped to a specific action
    • Expire after a set time
    • Cannot be reused
  • ✅ Good confirmation dialogs show: full content, target account/system, time, and platform — not just "are you sure?"
  • ✅ Application hooks can intercept tool calls before execution to enforce compliance rules

🧩 Scenario-Based Thinking

Scenario: An MCP server has a remove_member tool with a dry_run boolean for previewing before execution. Production monitoring shows the agent sometimes calls remove_member(dry_run=false) without first doing a preview run.

What is the best redesign?

  • A) Add a prompt instruction: "Always call with dry_run=true before dry_run=false"
  • B) Remove the dry_run parameter and add confirmation steps in the system prompt
  • C) Replace with two tools: preview_remove_member (returns impact details + confirmation token) and execute_remove_member (requires that token, bound to the specific previewed action)
  • D) Add an audit log so you can see when the agent skips the preview

Answer: C — The two-step pattern with confirmation tokens is the correct design. A dry_run boolean can be bypassed by the agent. Two separate tools with a token make bypassing structurally impossible — execute_remove_member cannot succeed without a valid token generated by preview_remove_member.

💡 Memory Tricks

Confirmation Token = OTP: Just as a banking OTP expires quickly and works only for the specific transaction you triggered, a confirmation token is time-limited and bound to the specific previewed action.

Application Layer = Seatbelt: You can tell someone "always wear a seatbelt" (prompt instruction), but a car that physically won't start unless the seatbelt is buckled (application enforcement) is infinitely more reliable.

❓ Chapter Practice Questions

Q1. An agent processes refunds. Company policy requires all refunds above ₹5,000 to have manager approval. Currently, this rule is stated in the system prompt: "Do not process refunds above ₹5,000 without manager approval." During an audit, you find the agent processed several large refunds without approval. What is the most effective fix?

  • A) Rewrite the system prompt to emphasise the rule more strongly
  • B) Add a few-shot example in the system prompt showing the correct escalation behaviour
  • C) Implement an application-layer hook that intercepts any process_refund call above ₹5,000, blocks it, and triggers the escalation workflow — regardless of what Claude was instructed
  • D) Lower the autonomous refund limit to ₹1,000 to reduce risk

Answer: C

Explanation: Compliance rules must be enforced at the application layer. Prompts are not reliable enforcement mechanisms — Claude may not follow them consistently, especially in complex multi-step conversations. An application hook that physically blocks the tool call and triggers escalation for amounts above the threshold is the only reliably compliant design.

Q2. A content publishing agent shows users a confirmation before posting. Users report they are frequently approving posts with incorrect accounts or wrong scheduled times because the confirmation dialog only says "Ready to publish. Confirm? (yes/no)". What change most effectively addresses this?

  • A) Require users to type the word CONFIRM instead of clicking yes
  • B) Add a 10-second delay before accepting confirmation to encourage review
  • C) Include the complete post content, target account, scheduled time, and platform in the confirmation request so users can verify all details before approving
  • D) Send the confirmation to a separate email instead of in-chat

Answer: C

Explanation: The problem is that users cannot verify correctness from the current vague confirmation. A confirmation is only useful if it shows all the specific details users need to check. Including the full content, target account, scheduled time, and platform allows users to catch errors before they happen.

Q3. Why is a confirmation token generated by a preview step more reliable than a simple confirmed: true boolean parameter in ensuring genuine human review?

  • A) Confirmation tokens use cryptographic hashing for security
  • B) A confirmed: true boolean can be passed by the agent at any time without completing a preview; a token can only be obtained by executing the preview step, and it is scoped to and expires after the specific previewed action
  • C) Confirmation tokens work across different sessions while booleans do not
  • D) Booleans add unnecessary complexity to tool schemas

Answer: B

Explanation: A confirmed: true boolean is just another parameter — the agent can set it to true without ever actually doing a preview. A confirmation token is physically generated by completing the preview step, is bound to the specific previewed action (not any action), and expires. This makes bypassing impossible at the application layer.

📌 Quick Revision Summary

  • Business rules must be enforced at the application layer — prompts alone are insufficient
  • Two-step pattern: preview tool (generates token) + execute tool (requires token)
  • Confirmation tokens: scoped to specific action, time-limited, single-use — better than dry_run boolean
  • Good confirmation dialogs show: full content + target account/system + scheduled time + platform
  • Application hooks intercept tool calls before execution — enforce compliance regardless of Claude's reasoning
  • Least privilege: give agents only the permissions they need for their specific tasks
